# 挑战数据审计

最后复核：2026-07-13。

## 已确认并修正

### 经典书单不是探索型目录

公开备份中的 `Top 150` 是一个完整书目清单，不按列表顺序逐级增加难度。现在采用图鉴型规则：

- 150 项全部为等级 1，并从一开始完整可见。
- 每本书都是终身一次成就。
- 全部使用统一的支线任务奖励档：145 XP、智慧 +5、文化 +2。
- 书单不使用分类迷雾、发现名额或 Tier 里程碑。
- 书单不会自动进入每日推荐；收藏或接取某本书后，才可能被推荐。
- 书单目录直接展示全部 150 项，并显示 `已读 / 150` 进度。

已有完成记录保存了当时的奖励快照，不会被本次规则修正改写。

### 中文一致性

- 任务标题统一使用“你”，不再混用“您”。
- 修正了“一份账单”的生硬语序。
- 修正“参见自由女神像”等直译表达。
- 修正“府慈山”为“富士山”。

## 仍需逐项复核

### P0：重复周期是目前最大的成就数据问题

原挑战备份没有携带完整周期信息，目前大量周期由标题关键词和任务位置推断。审计发现约 60 项被标为“每日”的任务包含参观、购买、旅行、加入团队、拍摄专题或完成作品等行为，明显可能更适合每周、每月或终身一次。

不能用一条简单规则批量修正，因为同一行为的合理周期取决于任务语义。例如“拍摄宠物”可以循环，而“加入一个团队”通常是终身一次。下一步应该为除经典书单和一生必去以外的 363 项任务建立人工周期表，并在修改前输出差异清单。

### P1：奖励仍受历史生成公式影响

挑战数据中的奖励是在不同开发阶段生成的静态值，而自定义任务使用当前统一奖励公式。两者的整体曲线接近，但并非完全同源。建议在周期人工复核完成后统一重新计算未完成任务奖励；历史完成记录继续使用奖励快照。

### P1：中文标题仍有宽泛或机器翻译痕迹

目前仍有约 18 个标题包含“做一些”“某事”“一些东西”等模糊表达。它们不一定错误，但行动标准不够清晰。建议逐项参考英文原文润色，同时避免给原挑战添加原本没有的硬性条件。

经典书单还需要一次独立的书目校对。目前作者和书名都写在单个标题字符串中，部分项目存在“《书名》作者：某人”、作者位置颠倒、非常用中文译名或同名书难以区分等问题。建议后续拆成结构化字段：

- `bookTitle`：规范中文书名。
- `bookTitleOriginal`：英文原名。
- `author`：规范作者名。
- `aliases`：其他常见译名。

这样可以按作者排序、搜索别名，并避免继续依赖机器翻译后的整句标题。

### P1：消费和捐赠任务需要更谨慎的推荐策略

约 15 项任务涉及购买、支付或捐赠。这些任务可以保留为原版内容，但不应频繁成为默认的“轻松一胜”。后续可以增加 `recommendationPolicy`：

- `normal`：正常进入每日推荐。
- `opt-in`：只有收藏或接取后才推荐。
- `never`：只在任务公会中主动查找。

经典书单已经先采用了类似 `opt-in` 的行为。

### P2：原站分类与备份分组并不完全相同

原站导航使用较大的分类，例如音乐和摄影曾归入更大的内容分类；挑战备份则提供 18 个更细分的标题分组。当前重制版保留 18 个分组，因为搜索和探索体验更清楚，但文档中应继续说明这是重制版的信息架构，而不是对原站导航的一比一复制。

### P2：部分挑战可能需要安全或时代背景提示

健康、理想体重、投资、器官捐赠、独自旅行等任务可能涉及身体、财务、法律或地区差异。任务本身可以作为原版档案保留，但描述可增加简短提示，例如“按自身情况并在需要时咨询专业人士”，而不是把它们包装成普适建议。

## 自动审计

运行：

```bash
npm run data:audit
```

自动检查 538 项数量、ID 和标题唯一性、中英文描述、18 个数据分组，以及经典书单的 150 项完整性、等级和周期规则。需要语义判断的问题只报告警告，不会被脚本擅自批量修改。
